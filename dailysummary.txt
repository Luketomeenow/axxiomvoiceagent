1. THE INCIDENT (Ameritex start, 2026-07-15 ~12:15 EDT)
=====================================================================
- Operator started Ameritex with concurrency 3 -> the dialer placed 3
  simultaneous calls to the SAME number (three lead rows, one contact),
  seconds apart. Confirmed in the data: the three 16:16Z calls all went
  to one +1301 number and died as stale-timeout.
- WHY: lead lists map ONE contact to MANY buildings. 186 numbers are
  duplicated within campaigns; one Ameritex number covers 11 lead rows.
  The dialer picked the top-N lead ROWS by score and its duplicate-call
  guard only checked per-LEAD, not per-PHONE.
- STUCK "RINGING": the stale-call sweeper only runs while the worker
  ticks, and the worker stopped the moment all campaigns paused — so a
  mid-flight call at pause time stayed "ringing" forever. On top, the
  dashboard End-call button gave up when Vapi's control URL timed out
  (the "End-call failed" errors in the logs) without clearing the row.

=====================================================================
2. DIALER FIXES (src/outbound/dialer.ts, routes.ts, index.ts)
=====================================================================
- Per-NUMBER in-flight guard in placeCall: a number with a live call can
  never be dialed again until it ends (applies to manual call-now too).
- Each campaign tick now fetches a wider candidate pool (slots*10, max
  200) and dedupes it to DISTINCT phone numbers -> concurrency 3-5 means
  3-5 different people. Skipped leads no longer consume a slot (only
  successful dials count against concurrency).
- Worker lifecycle: keeps ticking while ANY call is still live (even with
  zero running campaigns) so the sweeper can close strays; pause route +
  boot-resume updated to match (resumes after deploy if calls are live).
- End-call: on control-URL timeout/4xx (call usually already over on
  Vapi's side) the row is now STILL marked ended (ended_by=operator,
  reason "unconfirmed") so the monitor clears; a late end-of-call webhook
  overwrites with the real outcome if the call was somehow live.
- One-off cleanup: swept the call stuck "ringing" in prod DB (Ameritex,
  created 16:33Z) + returned its lead to a retryable disposition. Live
  calls now 0. Backend typecheck clean.
